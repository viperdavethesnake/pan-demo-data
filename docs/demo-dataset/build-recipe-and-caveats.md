# Build recipe & caveats

For the architect / demo operator: how this dataset was produced, how to reproduce it, and what must *not* be claimed about it on stage.

---

## Build recipe (2026-04-19 run)

The full build is orchestrated by `build-10M.ps1` at the repo root. It drives `PanzuraDemo v4.1.0` in five phases, then runs Orphanize + Report.

### Layered file-generation design

Four passes, each invoking `Invoke-DemoPipeline … -Phase Files` independently. Layering lets different date presets stack naturally on the same folder tree instead of one preset having to model the whole distribution.

| # | Preset | Files | Date window | RecentBias | Purpose |
|---|---|---:|---|---:|---|
| L1 | `LegacyMess` | 3,500,000 | -10 y → now | 70 | Old-skewed spread; 3-era weighting (40/30/30) avoids uniform plateau |
| L2 | `YearSpread` | 3,000,000 | -10 y → now | — | Uniform fill across full 10-year window |
| L3 | `RecentSkew` | 3,000,000 | -3 y → now (default) | 30 | Power-law tail skewed toward present |
| L4 | `Uniform` | 500,000 | **2019-01-01 → 2019-12-31** | — | **Deadbeat acquisition cohort** — visible 2019 bulge |

Totals: 10,000,000 planned · 9,962,001 created · 1,649 errors (0.016%).

### Why these knobs

- **10-year window on L1/L2** — default `MinDate` is `Now − 3y`. Extending to 10 y prevents a "3-year cliff" where preset-drawn files end abruptly at 3y and the Dormant/LegacyArchive classes (20% of files) pin to a uniform 3–5y block. With the extension, dormant ages blend smoothly with preset-drawn ages.
- **3-year window left default on L3** — `RecentSkew` is the "recent activity" layer; extending it would dilute the recent tail.
- **L4 500K / 2019-only** — small enough to not dominate the overall shape, tight enough to register as a visible bulge (9.63% of files vs. 3–4% in neighboring years).
- **No `-Parallel`** — `ForEach-Object -Parallel` is *slower* at this workload (decision #19 in `V4_SPEC.md`). Native sequential + P/Invoke SetNamedSecurityInfoW is the fastest path.

### Reproducing the build

```powershell
# 1. Clean slate
Import-Module 'C:\Users\Administrator\Documents\pan-demo-data\PanzuraDemo\PanzuraDemo.psd1' -Force
$cfg = Import-DemoConfig -Path default
Reset-DemoEnvironment -Config $cfg -IncludeShare -IncludeLegacyGroups -Confirm:$false
Get-ChildItem 'S:\Shared' -Force | ForEach-Object { Remove-Item $_.FullName -Recurse -Force }

# 2. Launch
pwsh -NoProfile -File 'C:\Users\Administrator\Documents\pan-demo-data\build-10M.ps1'
```

Expect ~8.5–9 hours wall clock on this host (16 CPU, S:\ NVMe Gen4 DRAM). See `build-10M.log` for phase timing from this run.

## Caveats — don't say these things on stage

### About the data itself

- ❌ **"This is 85 TB of real files."** Files are NTFS sparse — physical on-disk footprint is ~1.2 TB. The logical size is what a scanner sees and what ages / tiering decisions rightly act on, but don't claim it's physical storage.
- ❌ **"Here's a confidential HR document."** File *bodies* are random bytes + a correct magic-byte header for the extension. There is no real document content anywhere in the dataset. Don't open files on stage or imply document-level insight (search, summarization, content AI).
- ❌ **"These dates are authentic customer history."** Timestamps are synthetic — generated by a date-preset engine with coherent CT ≤ WT ≤ AT invariants. They *look* real (they pass contamination checks) but represent no actual business history.
- ❌ **"These user names are real."** 361 users are generated from first-name/last-name pools. The 40 "former employee" orphan SIDs map to deleted synthetic accounts (aaliyah.leon, christine.vo, etc.) — they aren't real people. Don't screenshot user lists next to customer logos.

### About the acquisition story

- ⚠️ **"Deadbeat Corp" is a demo narrative.** The 2019 file bulge is an engineered signal (500 K files created with `DatePreset=Uniform`, `MinDate=2019-01-01`, `MaxDate=2019-12-31`) chosen to give the demo a storytelling hook. There are no folder markers naming "Deadbeat," no unique AD group, no distinctive extensions — only the date cohort. If an analyst digs past the report into folder structure, the acquisition "evidence" doesn't extend beyond the CT distribution.
- ✅ That's sufficient for demo purposes (the report is what the audience sees), but don't promise "you can drill into the acquired data subtree" — there isn't one.

### About the AD environment

- ✅ AD is an isolated lab domain: `demo.panzura`, all demo artifacts under `OU=DemoCorp`. Not tied to any customer directory, not replicated to production.
- ❌ Don't show the literal host name (`PANZURA-SYM02`) or UNC path (`\\PANZURA-SYM02\Shared`) prominently — the demo pitch is generic "enterprise file share."

### About reproducibility

- ✅ The dataset is regeneratable from scratch in ~8.5 h with `build-10M.ps1`. Full cleanup + rebuild is a one-command flow.
- ⚠️ Year distribution **shifts as real time passes.** The Dormant / LegacyArchive classes hard-pin CT to `Now − 3y` … `Now − 5y` (see `New-DemoFile.ps1` lines 347–351). A build done today places dormant mass in 2021–2023. A build done in 6 months places it in 2021-late–2024-early. If the demo needs consistent-looking numbers over a long roadshow, re-run periodically or pin the host clock.

### About the ACL mess

- ✅ All ACL patterns (lazy GG, orphan SID, Everyone, Deny, inheritance breaks) are deliberately engineered to known ratios (`Mess.AclPatterns` in `default.psd1`). They represent realistic enterprise mess — not fabricated extremes.
- ⚠️ "Orphan SID" here means a SID whose AD account was deleted *after* file ownership was set. It's a real Windows-audit scenario, but do not claim these came from a breach or deprovisioning incident. They come from the `Remove-DemoOrphanUser` phase deleting flagged synthetic users.

## Where to go from here

- **Tweaking shape** (different date presets, different file counts, different dept ratios): edit `PanzuraDemo/config/default.psd1`, re-run `build-10M.ps1` after adjusting layer parameters.
- **Adding a new mess pattern** (e.g., stale shortcuts, deeply nested `__legacy__`, honey-trap file names): extend `PanzuraDemo/Private/Mess/` and wire into `New-DemoFolderTree` / `New-DemoFile`.
- **Changing the Deadbeat story** (different year, different cohort size, dedicated subtree): edit the L4 block in `build-10M.ps1`. Adding a dedicated subtree is the only part that needs code changes (a `-PathFilter` knob on `New-DemoFile` — see [`demo-narrative-and-widgets.md`](demo-narrative-and-widgets.md) notes on why we didn't do this).
- **Different scale** (1M for fast demos, 50M for scale-testing): change the `-MaxFiles` per layer proportionally. Current performance model: ~400 f/s cold → ~260 f/s dense.
